# 周报格式参考

这个参考文件固化周报的目标格式。它定义输出结构，不负责提供周报数据。

默认数据源改为 Git：

- 周报优先基于指定仓库的一周提交记录生成
- 可辅以工作区未提交改动判断未收尾事项
- 不默认依赖日报、项目总结或其他二次整理文档

## 核心要求

每周的进展和计划都按 `概述 + 明细` 的形式展开，并使用多级列表呈现：

- 顶层：有序列表
- 子层：无序列表
- 明细展开：继续使用无序子列表

## 列表结构

```markdown
1. [仓库名 / 主题名]
   - 本周进展概述：[一句话概述]
   - 本周进展明细：
     - [子模块 / 子主题]
       - [细节 1]
       - [细节 2]
   - 下周里程碑：[一句话概述]
   - 下周计划明细：
     - [具体工作项]
     - [具体工作项]
```

- 不要把正文写成纯段落
- 不要把正文写成 `##` 二级标题分节
- 要体现“有序列表嵌套无序列表”的层级关系
- 每个仓库都要完整包含这四项
- 不要把 `下周里程碑` 和 `下周计划明细` 聚合到正文最后

### 1. 本周进展概述

- 一句话概述
- 重点回答：
  - 什么事情做到了什么程度
  - 上了什么内容或能力
  - 当前使用、试用、验证或推广状态如何

### 2. 本周进展明细

- 展开模块进展、方案要点、过程问题、功能介绍等
- 优先按主题归组，不按时间顺序平铺

### 3. 下周里程碑

- 一句话概述
- 重点回答：
  - 下周最值得期待的结果是什么
  - 哪个关键节点最值得写出来

### 4. 下周计划明细

- 简单列具体工作项
- 保持可执行、可验证

## 改写要求

- 原始日报或项目总结不符合这个结构时，必须重组后再输出
- 原始输入如果是 Git 提交记录，也必须先按主题归组，再输出
- 原始材料写得很散时，先按主题合并，再写概述和明细
- 没有证据支持的“使用情况”“上线效果”不能编造
- 没有明确下周计划时，可以从未完成事项中提炼计划，但要保持为计划表述
- 如果同一周内存在多个主题，优先拆成多个顶层有序列表项
- 下周里程碑要按仓库分别提炼
- 下周计划要按仓库分别列出
- 用户指定了仓库范围时，只能写这些仓库的内容

## 适用说明

- 这个格式默认用于 `陈洁` 的周报
- 外部语雀模板可以作为辅助参考
- 如果外部模板页面无法访问，仍然以本文件的格式要求为准
